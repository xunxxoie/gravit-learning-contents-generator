## 모범 예시 (few-shot)

생성기는 이 예시를 본보기로 삼아 같은 품질의 문제를 만든다. 형태를 그대로 베끼라는 뜻이 아니라, 발문·본문·선지·해설이 맞물리는 방식을 따르라는 뜻이다. 본문 유형(개념 설명 문장·코드·계산 전제·표·상황)별로 예시를 하나씩 두어, 한 형태만 흉내 내다 본문 다양성이 죽는 일을 막는다. 형태 자체는 본보기일 뿐이니 새 주제에 맞게 변형하라.

---

### 예시 1 — 개념을 판별하는 객관식

- **발문(instruction):** **아래 커널 구조에 대한 설명으로 옳은 것은?**
- **본문(content):** **마이크로커널은 스케줄링·IPC 등 최소한의 기능만 커널 공간에 두고, 파일 시스템·디바이스 드라이버 등은 사용자 공간의 독립된 서비스로 분리한 구조이다.**
- **선지(option):**

| 선지 내용 | is_answer | explanation |
|---|---|---|
| 서비스 하나가 비정상 종료되면 시스템 전체가 곧바로 패닉에 빠진다. | false | 모놀리식 커널의 약점을 갖다 붙인 오개념이다. 마이크로커널은 서비스가 분리돼 있어 한 서비스의 장애에 오히려 강하다. |
| 사용자 공간 서비스 사이의 요청이 IPC를 거쳐, 모놀리식 커널보다 통신 오버헤드가 크다. | true | 기능을 분리한 구조에서 치르는 대가다. 최소한의 커널이 갖는 핵심 트레이드오프다. |
| 디바이스 드라이버가 모두 커널 모드에서 실행되어 모드 전환이 없다. | false | 모놀리식 커널의 동작을 갖다 붙인 오개념이다. 드라이버가 사용자 공간에 있어 모드 전환이 일어난다. |
| 커널 코드 양이 많아 다른 환경으로 옮기기 어렵고 유지보수가 힘들다. | false | 모놀리식 커널의 단점을 거꾸로 갖다 붙인 오개념이다. 커널이 작아 오히려 유리하다. |

**왜 좋은가:** 본문을 지우면 정답(IPC 오버헤드)과 오답(이식성)이 둘 다 "맞는 말"이 되어 정답이 하나로 정해지지 않는다 — 즉 본문이 정답을 정한다(INV-1). 선지는 본문에 적힌 **최소 기능·분리**를 되풀이하지 않고 IPC·장애 격리·이식성 같은 다른 특징을 다룬다. 오답 셋은 모두 모놀리식 커널의 특징을 잘못 갖다 붙인 것이고, 해설이 그 오개념을 짚는다.

### 예시 2 — 자료를 해석하는 객관식

- **발문(instruction):** **아래 트랜잭션을 실행한 뒤 컬럼 v의 최종 값은?**
- **본문(content):** 아래 코드 블록(마크다운)

```sql
-- 초기 상태: v = 10
UPDATE t SET v = 20 WHERE id = 1;
SAVEPOINT sp1;
UPDATE t SET v = 30 WHERE id = 1;
ROLLBACK TO sp1;
COMMIT;
```

- **선지(option):**

| 선지 내용 | is_answer | explanation |
|---|---|---|
| 10 | false | **ROLLBACK TO sp1**을 전체 롤백으로 오해한 것이다. sp1 이전의 **UPDATE**(v=20)는 그대로 남는다. |
| 20 | true | **ROLLBACK TO sp1**은 sp1 이후의 변경(v=30)만 취소한다. sp1 이전의 v=20이 남고 **COMMIT**으로 확정된다. |
| 30 | false | **ROLLBACK TO sp1**을 무시하고 마지막 **UPDATE**가 그대로 반영된다고 오해한 것이다. |
| null | false | 롤백이 행이나 값을 지운다고 오해한 것이다. **ROLLBACK TO**는 행을 지우지 않는다. |

**왜 좋은가:** 본문(코드)이 없으면 풀 수 없다(INV-1). 선지가 모두 **값** 한 가지 기준 위에 있다(INV-5). 오답 셋이 저마다 다른 트랜잭션 오개념을 담고, 해설이 그것을 짚는다(EXP-2). 본문을 코드 블록(마크다운)으로 쓴 예이기도 하다.

### 예시 3 — 단답형 주관식

- **발문(instruction):** **아래 설명에 해당하는 TCP 혼잡 제어 단계의 이름은?**
- **본문(content):** **매 RTT마다 cwnd(혼잡 윈도)를 두 배로 늘려, 사용할 수 있는 대역폭을 빠르게 찾아 가는 초기 단계.**
- **정답(answer.content):** **슬로 스타트,slow start,슬로우 스타트,느린 시작**
- **해설(answer.explanation):** **cwnd를 지수적으로 늘리는 초기 단계가 슬로 스타트다. cwnd를 선형으로 천천히 늘리는 혼잡 회피(congestion avoidance) 단계와 헷갈리지 않도록 주의한다.**

**왜 좋은가:** 정답이 짧은 용어로 떨어지고, 한글·영문·표기 변형을 정답 목록에 빠짐없이 모았다(INV-3). 해설이 정답의 근거와 함께, 헷갈리기 쉬운 옆 단계(혼잡 회피)와의 경계를 짚는다(EXP-4).

### 예시 4 — 계산 전제를 푸는 객관식

- **발문(instruction):** **아래 조건에서 단일 단계 페이지 테이블의 엔트리 개수는?**
- **본문(content):** **가상 주소 공간은 32비트이고, 페이지 크기는 4KB이다.**
- **선지(option):**

| 선지 내용 | is_answer | explanation |
|---|---|---|
| 4,096 | false | 페이지 오프셋 비트 수(12)를 엔트리 개수로 혼동한 것이다. 오프셋은 페이지 **내부** 위치를 가리킬 뿐 엔트리 수가 아니다. |
| 1,048,576 | true | 페이지 번호 비트 수 = 32 − 12 = 20이므로 엔트리 수 = 2²⁰ = 1,048,576이다. 4KB = 2¹²라 오프셋이 12비트다. |
| 4,194,304 | false | 페이지 크기를 1KB로 착각해 오프셋을 10비트로 본 것이다(2²²). 4KB면 오프셋은 12비트다. |
| 4,294,967,296 | false | 페이지 크기로 나누지 않고 주소 공간 전체(2³² 바이트)를 그대로 엔트리 수로 본 것이다. |

**왜 좋은가:** 본문이 계산에 필요한 전제(주소 비트 수 **+** 페이지 크기)를 모두 주어 정답이 하나로 떨어진다(INV-6). 본문을 지우면 풀 수 없다(INV-1). 선지가 모두 **개수** 한 단위 위에 있고(INV-5), 오답 셋이 저마다 다른 계산 오개념(오프셋 혼동·페이지 크기 착각·나눗셈 누락)을 담아 해설이 그것을 짚는다(EXP-2). 본문을 계산 전제로 쓴 예다.

### 예시 5 — 표를 해석하는 객관식 (부정 발문)

- **발문(instruction):** **위 비교표를 바탕으로 옳지 않은 것은?**
- **본문(content):** 아래 표(마크다운)

| 알고리즘 | 선점 여부 | 선택 기준 |
|---|---|---|
| FCFS | 비선점 | 도착 순서 |
| SJF | 비선점 | 실행 시간이 짧은 작업 |
| RR | 선점 | 타임 퀀텀 단위 순환 |

- **선지(option):**

| 선지 내용 | is_answer | explanation |
|---|---|---|
| FCFS는 비선점이라, 앞선 긴 작업이 뒤의 짧은 작업을 오래 기다리게 하는 호위 효과가 생길 수 있다. | false | 비선점 + 도착 순서라는 표의 내용에서 바로 따라 나오는 참인 진술이다. |
| SJF는 실행 시간이 짧은 작업을 먼저 처리하므로 평균 대기 시간을 줄이는 데 유리하다. | false | 선택 기준이 짧은 작업 우선이므로 참이다. SJF가 평균 대기 시간 최소화에 유리하다는 성질과 맞다. |
| RR은 타임 퀀텀이 끝나면 실행 중인 작업을 준비 큐로 돌려보내고 다음 작업에 CPU를 넘긴다. | false | 선점 + 타임 퀀텀 순환이라는 표의 내용과 일치하는 참인 진술이다. |
| RR은 비선점이라 한 작업이 끝날 때까지 CPU를 놓지 않는다. | true | 표에서 RR은 **선점**인데 이를 비선점이라고 한 거짓 진술이다. 타임 퀀텀으로 CPU를 강제 회수하는 것이 RR의 핵심이다. |

**왜 좋은가:** 본문이 표(마크다운)인 예다. 발문이 부정형(**옳지 않은 것은?**)이라 정답은 표와 모순되는 거짓 진술 1개이고, 오답 셋은 표에서 타당하게 도출되는 참인 진술이다(INV-4 부정 발문 규칙). 정답 진술의 진위가 표의 **RR=선점**에 직접 걸려 본문이 일을 한다(INV-1). **is_answer=true**가 "옳은 선지"가 아니라 "발문이 고르라는 선지(=틀린 진술)"임에 주의한다.

### 예시 6 — 상황·사례를 푸는 주관식

- **발문(instruction):** **아래 상황에서 발생한 문제를 가리키는 용어는?**
- **본문(content):** **두 스레드 A와 B가 공유 변수 counter를 각각 1씩 증가시킨다. 둘 다 counter를 읽은 뒤 1을 더해 다시 쓰는데, 실행이 엇갈리면 두 번 증가시켰는데도 최종값이 1만 늘어난다.**
- **정답(answer.content):** **경쟁 상태,경쟁 조건,race condition,레이스 컨디션**
- **해설(answer.explanation):** **공유 자원에 대한 접근이 동기화되지 않아 실행 순서(인터리빙)에 따라 결과가 달라지는 것이 경쟁 상태다. 자원을 서로 기다리며 멈추는 교착 상태(deadlock)나, 보호해야 할 코드 영역을 가리키는 임계 구역(critical section)과 헷갈리지 않도록 주의한다.**

**왜 좋은가:** 본문이 상황·사례인 예다. 상황을 지우면 무엇을 묻는지 알 수 없어 본문이 일을 한다(INV-1). 정답이 짧은 용어로 떨어지고 한글·영문·표기 변형을 빠짐없이 모았다(INV-3 주관식). 해설이 정답의 근거와 함께 헷갈리기 쉬운 옆 개념(교착 상태·임계 구역)과의 경계를 짚는다(EXP-4).
